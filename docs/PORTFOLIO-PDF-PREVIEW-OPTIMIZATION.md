# 포트폴리오: PDF 미리보기 성능 최적화 (페이지 분할 + 다중 프로세스)

> 대용량 PDF 미리보기 응답 시간을 **기준선 측정 → 원인 분석 → 페이지 단위 분할·병렬 처리**로 개선한 기술 도전 정리.

---

## 1. Background (배경)

- **서비스**: PDF-Utils는 PDF 병합·분리·압축·미리보기 등 8개 기능을 제공하는 웹 툴이다.
- **미리보기 동작**: 사용자가 PDF를 업로드하면 `/api/pdf-preview` 가 **Ghostscript(GS)** 로 각 페이지를 이미지로 변환해 반환한다.
- **현재 구현**: 한 요청당 GS를 **페이지 수만큼 순차 호출**하거나, 전체를 한 번에 처리한다. GS는 **Node와 별도 프로세스**로 실행되며, Node는 I/O 대기만 하고 CPU 작업은 GS가 담당한다.
- **측정 인프라**: Locust 부하테스트 + CI에서 결과를 `POST /api/admin/load-test-results` 로 전송해 **load_test_results** 테이블에 적재하고, 관리자 대시보드(필터·정렬·대상 환경 로컬/배포)에서 조회할 수 있다.

---

## 2. Problem (문제)

- **증상**: 10MB·12페이지급 PDF 한 건당 미리보기 응답 시간이 **약 30초** 수준으로 길다.
- **원인**: 
  - GS가 **한 요청 안에서 페이지 전체를 순차 처리**하거나, 한 번에 무거운 작업을 수행한다.
  - **한 스레드(이벤트 루프)가 여러 요청을 받을 수는 있지만**, 개별 요청의 완료 시간은 GS 1회 처리 시간에 묶여 있다.
- **결과**: 사용자 체감 지연이 크고, 동시 접속이 늘어나면 전체 처리량도 GS 처리 시간에 의해 제한된다.

**Before (현재 구조)**  
- 요청 1건 → GS가 N페이지 전체 순차 처리 → 약 30초 후 응답.  
- 100페이지가 들어오면 이론상 더 길어질 수 있는 구조.

---

## 3. Solution (해결 방향)

**아이디어**: 페이지를 **인덱스 기준으로 chunk(예: 10페이지 단위)** 로 나누고, 각 chunk를 **독립된 GS 작업(다중 프로세스)** 로 병렬 실행한 뒤, **돌아오는 순서와 관계없이 페이지 인덱스로 정렬해 합쳐서** 클라이언트에 제공한다.

**After (개선 구조)**  
- 요청 1건(100페이지) → 10개 chunk(1–10, 11–20, …) 생성.  
- 각 chunk를 **워커/큐**를 통해 **동시에** GS로 처리 (예: `-dFirstPage=1 -dLastPage=10` 등).  
- 완료된 chunk 결과를 **페이지 번호 기준으로 병합** → 클라이언트에는 기존과 동일한 형식(페이지 순서 이미지 목록)으로 전달.  
- **기대 효과**: 전체 소요 시간이 “가장 긴 chunk 시간 + 병합 오버헤드” 수준으로 줄어들어, 순차 처리 대비 체감 지연 감소.

(구현 시 고려: 동시 GS 프로세스 수 제한, 실패한 chunk만 재시도, 캐시 연동 등.)

---

## 4. 측정·기록 전략 (포트폴리오용)

### 4.1 기준선(Before) 확보

- **목표**: “개선 전” 수치를 **배포 환경** 기준으로 남겨 두어, 포트폴리오에서 신뢰도 있게 보여준다.
- **방법**  
  1. **배포**: 현재 코드로 백엔드/프론트 배포 (EC2 스테이징 + Vercel 등).  
  2. **부하테스트 실행 (배포 대상)**  
     - **옵션 A (권장)**: CI 워크플로에서 이미 배포 후 Locust를 **배포 URL(EC2_BACKEND_URL)** 대상으로 돌리고, 결과가 **배포 DB**에 저장되므로, **배포된 관리자 페이지**에서 “대상 환경: 배포”로 필터 걸어 **캡처**.  
     - **옵션 B**: EC2에 SSH 접속 후, 같은 레포의 `backend/loadtest` 에서 Locust + `send_results_to_api.py` 를 **배포 URL**로 실행해 결과를 배포 DB에 넣고, 동일하게 관리자 페이지(배포)에서 캡처.  
  3. **캡처 내용**  
     - 관리자 > 부하테스트 결과: **대상 환경 = 배포**, 기간 = 전체.  
     - 테이블에 **평균(ms)**, **P95(ms)**, **파일 크기**, **총 요청** 등이 보이도록 스크린샷.  
     - (선택) 요청 로그 탭에서 `/api/pdf-preview` 1건의 **elapsed(ms)** 가 30초대인 로그 한 줄.

→ 이걸 **“Before: 배포 환경 기준선”** 으로 문서/포트폴리오에 사용.

### 4.2 개선 후(After) 확보

- **방법**: 페이지 분할 + 다중 프로세스(또는 워커/큐) 적용 후, **동일한 PDF·동일한 Locust 설정**으로 다시 부하테스트를 배포 대상으로 실행.  
- **캡처**: 관리자 > 부하테스트 결과(배포)에서 **개선된 평균/P95** 가 보이는 화면.  
- **정리**: Before 스크린샷 vs After 스크린샷을 나란히 두고, “페이지 분할·병렬 처리 도입으로 평균 N초 → M초” 식으로 요약.

### 4.3 로컬만 있는 경우

- 지금처럼 **로컬 DB에만** 결과가 쌓여 있다면, 포트폴리오에서는 “로컬 환경 기준선”으로 명시하고, “배포 환경에서도 동일한 트렌드로 개선할 수 있음을 목표로 했다”고 보완하면 된다.  
- 가능하면 **한 번은 배포 후 CI 또는 EC2 SSH로 배포 URL 대상 테스트**를 돌려, “배포” 필터로 나오는 수치를 Before로 쓰는 것을 권장한다.

---

## 5. 포트폴리오에 넣을 캡처·문구 요약

| 구분 | 내용 |
|------|------|
| **Background** | PDF 미리보기 = Ghostscript, 현재는 페이지 순차/일괄 처리. 부하테스트·관리자 대시보드로 수치 수집. |
| **Problem** | 10MB·12페이지 1건당 약 30초. 원인: GS 단일 흐름에 의한 지연. |
| **Solution** | 페이지를 chunk로 나누고, chunk 단위로 GS 다중 프로세스 병렬 실행 후 인덱스로 병합. |
| **Before 캡처** | 관리자 > 부하테스트 결과(대상 환경: 배포), 평균 30초대, 파일 크기 10MB대. |
| **After 캡처** | 동일 화면에서 개선된 평균/P95 (구현 후). |
| **다이어그램** | Node 단일 프로세스 ↔ GS 다중 프로세스, chunk 분할·병합 흐름 (텍스트/이미지). |

이 순서로 정리하면, “비동기 race condition + reqId” 예시처럼 **Background → Problem → Solution(Before/After)** 구조로 포트폴리오에 올릴 수 있다.
